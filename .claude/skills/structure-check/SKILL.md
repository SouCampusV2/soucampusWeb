# Structure Check

Проверка структуры проекта и архитектурных паттернов на консистентность — не про цвета/вёрстку (это `design-check`), а про то, как организован код, данные и страницы.

## Порядок действий

1. Прочитать `CLAUDE.md` целиком (если ещё не прочитан в этой сессии) — правила ниже кратко продублированы, но `CLAUDE.md` первичен.
2. Пройтись по каждому пункту чек-листа ниже, читая реальные файлы (не по памяти).

## Чек-лист

### 1. Форма данных ("table-shaped" данные)

Цель: данные, которые в будущем переедут в Supabase, уже сейчас должны быть оформлены как плоские массивы объектов, повторяющие структуру будущей таблицы (эталон — `ClientReviews.tsx`).

- `src/lib/projects.ts` — каждый проект: единообразные поля у всех объектов (`id`/`slug`/`title`/`tag`/`summary`/`description`/`image`/`price`/`size`/`deadline`/опционально `gallery`), ни один объект не пропускает обязательные поля молча.
- `Stats.tsx` — цифры оформлены как массив объектов с явными полями (`label`/`value`/`suffix` и т.п.), а не хардкод чисел прямо в JSX.
- `CountriesMarquee` — список стран/флагов оформлен так же структурно (массив объектов), даже если остаётся локальным (не переезжает в БД).
- `ClientReviews.tsx` — остаётся эталоном: `name`/`role`/`flag`/`text`/`accent`. Если менялась — не потеряла эту форму.

Если где-то данные вставлены как одиночные строки/числа прямо в разметку компонента вместо вынесенного массива — это нарушение.

### 2. Консистентность компонентов-паттернов

- Все кнопки — через `src/components/Button.tsx`, не пересобраны вручную (пересекается с `design-check`, но здесь смотрим именно на **архитектурный** повтор: не завёлся ли второй "почти такой же" компонент-кнопка в другом месте).
- Hero-секции всех страниц следуют единому паттерну (шрифт `Unbounded`, `pt-20`, радиальный градиент) — см. `DESIGN.md` → "Hero-секции страниц". Если появилась новая страница с Hero — сверить, что не изобретён новый вариант.
- Анимация перехода между страницами (`PageTransition`) подключена только через навбар-ссылки — если где-то появилась новая ссылка на внутреннюю страницу, проверить, не потеряна/не задублирована ли логика перехода.
- Иконки — только `@phosphor-icons/react`, не самодельные SVG-компоненты, не текстовые эмодзи-заглушки.

### 3. Структура страниц и роутинга

- Страницы лежат в `src/app/` по паттерну App Router (`page.tsx`, при необходимости `[slug]/page.tsx`), нет случайных дублей роутов под разными путями на один и тот же контент.
- `src/app/portfolio/[slug]/page.tsx` — динамический роут остаётся единой точкой рендера карточки проекта; если для новых работ понадобился особый layout — проверить, что это расширение общего компонента (например, через опциональное поле типа `gallery`), а не форк отдельной страницы под конкретный проект.
- Общие для всех страниц блоки (`Navbar`, `PricingSection`, `ContactFooter`) подключены один раз в `src/app/layout.tsx`, не задублированы внутри отдельных `page.tsx`.

### 4. Разделение переиспользуемого и одноразового

- Компонент в `src/components/` — если используется только в одном месте и не задуман как переиспользуемый, задать вопрос: не стоило ли инлайнить его в страницу вместо вынесения (излишняя абстракция) — но не переусердствовать, если у компонента уже понятная единственная ответственность.
- Если один и тот же кусок JSX/логики скопирован в 2+ местах почти без изменений — кандидат на вынос в общий компонент; отметить, но не переписывать без подтверждения.

### 5. Соответствие текущей фазе плана

- Не появилось ли реализации будущих этапов раньше времени (например, куски Supabase/Stripe-интеграции до Этапа 3/4) — если появилось, не обязательно ошибка, но стоит явно отметить и сверить с `CLAUDE.md`.
- Плейсхолдеры, перечисленные в CLAUDE.md → "Осталось как плейсхолдер", всё ещё присутствуют как плейсхолдеры (не были случайно закоммичены как финальные без ведома пользователя).

### 6. Дублирование функционала и компонентов

- Два и более компонента в `src/components/` (или разных страницах) решают одну и ту же задачу почти одинаковым кодом (например, повторно собранная карточка проекта, повторная карусель, повторный набор анимационных пропсов motion) — вместо переиспользования уже существующего компонента.
- Повторяющаяся логика (не только JSX-разметка) — например, одинаковый расчёт/форматирование данных, скопированный в нескольких файлах вместо вынесения в общую функцию/хук.
- Один и тот же тип данных описан разными наборами полей в разных местах (например, "проект" в `projects.ts` и отдельная похожая структура где-то ещё) — если такое найдено, явно указать оба места и предложить объединить.
- Не путать с осознанными "разными вариантами отображения карточки" (см. план по портфолио в CLAUDE.md — там разные варианты сделаны намеренно) — дублирование это когда копипаст, а не когда вариативность заложена сознательно.

## Формат ответа

Кратко, по пунктам — только реальные находки (если по пункту чек-листа всё в порядке, не расписывать — одна строка "нарушений не найдено" по итогу, если нарушений нет вообще). На каждую находку:

- файл (и строка, если применимо);
- что не так;
- какой пункт чек-листа / правило `CLAUDE.md` затронуто;
- конкретное предложение (что именно поменять), без немедленного исправления файлов — только отчёт, если явно не попросили исправить.
