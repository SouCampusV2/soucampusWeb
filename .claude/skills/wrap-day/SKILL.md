# Wrap Day

Ритуал конца сессии. Не спрашивать, что именно записать в документацию — самому проанализировать разговор и рабочее дерево.

## Порядок действий

1. **Просмотреть, что реально изменилось в этой сессии** — `git status`, `git diff`, и то, что обсуждалось/делалось в разговоре (новые фичи, деплой, контент, инфраструктурные решения, важные договорённости с пользователем).
2. **Обновить `CLAUDE.md`**, если есть что записать:
   - Раздел "Статус / прогресс" → "Сделано" — добавить, что появилось нового.
   - "Осталось как плейсхолдер" — вычеркнуть закрытое, добавить новое, если появилось.
   - "Следующий шаг по сайту" / "Ближайший план обучения" — подвинуть отметки ✅/⏳, поправить актуальный шаг.
   - Новые важные решения (архитектурные, workflow — например git-ветки, деплой, выбор технологии) — задокументировать там, где они логически относятся, а не только в "Сделано".
   - Не переписывать то, что не менялось — точечные правки, не полный рерайт файла.
3. **Проверить `README.md` — обязательный шаг, не пропускать молча.** После каждого обновления `CLAUDE.md` явно свериться, не разошёлся ли `README.md`:
   - `## Roadmap` — если пункт закрыт в CLAUDE.md/в этой сессии, отметить `~~зачёркнуто~~ — done` и в README тоже, не только в CLAUDE.md.
   - `## Stack` — новая зависимость/сервис, подключённые в этой сессии (см. `package.json`/`git diff`), должны попасть сюда (из "Planned" в "In use" или как новая строка).
   - `## Project structure` — новый файл/папка верхнего уровня, важная для внешнего читателя (например `RESPONSIVE_PLAN.md`), — добавить строкой.
   - README короче CLAUDE.md и для внешнего взгляда на проект — не дублировать туда внутренние детали/историю решений, только факт "что сейчас в проекте".
   - Если по итогам проверки менять нечего — это нормальный исход, но сама проверка должна произойти, а не быть пропущена по умолчанию.
4. **Проверить линт** (`npm run lint`), если правились файлы кода в этой сессии.
5. **Закоммитить и запушить**:
   - Проверить текущую ветку (`git branch --show-current`) — коммитить в неё же, не переключаться самовольно на `master`.
   - Один осмысленный коммит (или несколько логических, если изменения явно разнородные) с сообщением, отражающим суть, без AI co-authorship (см. правило в CLAUDE.md — автор коммитов только пользователь).
   - `git push`.
   - **Мёрдж в `master` — не часть этого ритуала.** Коммит и пуш идут в `dev` и на этом `wrap-day` заканчивается. Мёрдж в `master` — отдельное, осознанное действие пользователя, только после того, как он сам мини-протестировал изменения на Vercel Preview-ссылке (`https://soucampus-builds-git-dev-sou-campus.vercel.app`). Не предлагать и тем более не выполнять мёрдж автоматически в рамках `wrap-day`.
6. **Выдать саммари пользователю** — коротко, по делу:
   - Что делали в этой сессии (буллеты, сгруппировано по смыслу, не хронологический список каждого шага).
   - Что осталось открытым / что дальше по плану (взять из обновлённого CLAUDE.md, не выдумывать заново).
   - Не расписывать процесс — только результат и next steps.

## Важно

- Если сессия не содержала значимых изменений (только обсуждение без кода/контента) — не форсировать пустой коммит, сказать об этом прямо.
- Если рабочее дерево грязное по причине, не связанной с этой сессией (чужие правки, конфликты) — не коммитить молча, спросить пользователя.
- Не переписывать историю (`git commit --amend`, force push) — только новые коммиты поверх текущей ветки.
