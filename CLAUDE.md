Порядок именно такой, шаг за шагом, не перескакивать:

3. **После этого** — GitHub-репозиторий, CI/CD (по шагам, не всё сразу) и облачные агенты Claude Code (routines). ✅ Сделано — репозиторий подключён (Claude GitHub App, доступ через "Installed GitHub Apps", не через "Authorized OAuth Apps" — это разные вещи), CI (GitHub Actions: lint+build) настроен раньше, и настроен еженедельный routine (`design-check` + `structure-check` + отчёт в `reports/` + отправка файлом в Telegram через бота). ✅ Разобрали подробно, как облачные агенты (routines/CCR) устроены "под капотом" (изоляция/контейнеры, agent loop оркестратор↔модель, токены/allowed_tools, TCP/TLS/HTTP/git smart protocol, cron dispatcher, security-классификатор действий). См. также план проверки безопасности/производительности, отложенный ранее до появления репозитория — теперь репозиторий есть, можно вернуться к этому пункту при следующем подходящем моменте.

**Идея на будущее — переосмыслить структуру еженедельных отчётов (`reports/`), не только формат.** Сейчас у отчёта фиксированные 4 раздела (Дизайн-чек / Проверка структуры / Изменения за неделю / Термины и технологии) — пользователь хочет отдельно продумать, что вообще должно быть в отчёте по содержанию/структуре (не путать с визуальным форматированием — это уже сделано, см. `design-check`/`structure-check` промпт routine). Не реализовывать сейчас, вернуться к этому осознанно.

- Next.js и App Router
- TypeScript basics
- Tailwind CSS basics
- Git и GitHub
- базовые принципы UI/UX
- деплой через Vercel
- работу с переменными окружения и секретами
- базовые принципы безопасности

### Как работать с ИИ

- разбивать большие задачи на маленькие;
- задавать конкретные вопросы;
- не бояться переспросить;
- просить объяснение архитектуры и причин выбора решений;
- проверять результат, а не blindly копировать код.

## Правила для проекта

- Использовать простые решения там, где это уместно.
- Не усложнять архитектуру без причины.
- Сначала сделать рабочую версию, потом улучшать.
- Хранить секреты в переменных окружения.
- Не публиковать токены и ключи в репозитории.
- Делать изменения постепенно и документировать их.
- Git-репозиторий уже создан локально (Этап 1); автор коммитов — только пользователь, без соавторства ИИ в коммитах.
- Приоритет производительности: сайт должен грузиться максимально быстро (оптимизация картинок, минимум лишнего JS на клиенте — см. Server/Client Components).
- Для медленных/асинхронных блоков (загрузка данных, изображений) — использовать skeleton-плейсхолдеры вместо пустого экрана или спиннеров.

## Идея на будущее: подписка на постройки (Patreon-модель внутри магазина)

Помимо разовых заказов карт на заказ, в магазине (Этап 4) рассмотреть модель подписки в духе Patreon:

- несколько платных тарифов (черновик — см. `PricingSection.tsx` на главной, UI уже сделан как драфт);
- подписчики получают доступ к библиотеке готовых построек, которые автор выкладывает регулярно;
- полученные по подписке постройки закрепляются в личном профиле пользователя на сайте (то есть нужен профиль/аккаунт пользователя — завязано на Supabase Auth, Этап 4);
- более высокие тарифы — дополнительные права (например, коммерческая лицензия на использование построек).

Статус: пока только идея и готовый UI-черновик секции на главной странице. Логика подписок, оплата, привязка построек к профилю — не реализованы, планировать при подходе к Этапу 4 (магазин + авторизация).

## Идея на будущее: аналитика/трекинг посетителей

Отслеживание посещаемости и поведения на сайте (сколько заходит, какие страницы смотрят, что кликают).

- **Сделано**: подключён **Vercel Analytics** (`@vercel/analytics`, компонент `<Analytics />` в `src/app/layout.tsx`) — раз сайт и так деплоится на Vercel, самый простой вариант для старта. Показывает посещения страниц, источники трафика, устройства. Данные видны только после деплоя (в дашборде Vercel → Analytics), локально в dev ничего не собирает. Бесплатный лимит — 2500 событий/мес (событие ≈ один просмотр страницы), после лимита сбор данных на месяц останавливается, сайт при этом продолжает работать нормально.
- **Следующий шаг (позже, ориентир — перед Этапом 4/магазином, когда портфолио-сайт уже готов и есть реальные посетители)**: добавить более детальный трекинг поведения — что именно кликают, тепловые карты, записи сессий, воронки покупки. Рассматривать **PostHog** (из коробки даёт heatmaps и session replay, не нужно вручную помечать каждую кнопку событием) как более продвинутую альтернативу/дополнение к Vercel Analytics — особенно ценен на этапе магазина, где важны сценарии "каталог → товар → покупка". Cookie-consent баннер по возможности не усложнять, выбирать privacy-friendly решения (Vercel Analytics/PostHog в базовой конфигурации обычно не требуют баннера).

Если потребуется, этот план можно будет менять по ходу работы.
