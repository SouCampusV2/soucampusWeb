Обновлять `DESIGN.md`, а не этот файл, когда меняется именно дизайн-система (новый цвет, новое правило вёрстки, новый переиспользуемый компонент).

## Текущий приоритет (обучение + следующий шаг по сайту)

Этот блок — единственное место, откуда стоит начинать чтение файла: здесь и порядок обучения, и конкретные ближайшие шаги по сайту. Всё остальное ниже (контент, дизайн, Этапы 1-5, идеи на будущее) — по-прежнему в силе, но это справочная часть, не то, с чего начинать сессию.

### Ближайший план обучения (приоритет выше остального контента/дизайна)

Порядок именно такой, шаг за шагом, не перескакивать:

1. **Разобраться в функционале Claude Code** — что уже встроено из коробки: слэш-команды, инструменты, режимы, память и т.п. Пользователь хочет пройтись по этому вместе, разобраться что есть и как этим пользоваться. ✅ Прошли.
2. **Skills** — как писать свои кастомные навыки (`.claude/skills/`), потренироваться сделать один-два под этот проект. ✅ Начали — есть первый скилл `design-check` (`.claude/skills/design-check/SKILL.md`), уже опробован на проверке типографики сайта.
3. **После этого** — GitHub-репозиторий, CI/CD (по шагам, не всё сразу) и облачные агенты Claude Code (routines) — см. также план проверки безопасности/производительности, отложенный до появления репозитория. ⏳ В процессе — настроен еженедельный routine (`design-check` + `structure-check` + отчёт в `reports/` + отправка в Telegram). Отдельно — пользователь хочет разобраться подробнее, как облачные агенты (routines/CCR) устроены "под капотом" (изоляция, environment, откуда берётся доступ к репозиторию и т.д.), не только как их настраивать через UI/API.
4. **Дальше** — возвращаемся к контенту и улучшению дизайна (см. плейсхолдеры выше и "Следующий шаг по сайту" ниже).
5. **Потом** — спринты, работа с БД (Supabase, см. Этап 3) и далее по плану развития проекта.

**"Graphify" — отложено.** Идея пользователя разложить файлы/данные/правила/скрипты проекта на структурированную систему заметок (в духе Obsidian, markdown + `[[wikilink]]`-связи) ради структурированной документации и экономии токенов. Пользователь хочет сначала сам изучить тему подробнее и вернуться с более чётким запросом — пока не встраивать в план и не трогать `CLAUDE.md`/`DESIGN.md` под это.

## Цель проекта

## Рекомендуемый стек на старте

### Frontend

- Next.js
- TypeScript
- Tailwind CSS

### Backend / данные

- Supabase как простой и быстрый вариант на старте
- PostgreSQL как более зрелый вариант в будущем

### Авторизация / пользователи

- Решено: **Supabase Auth** (не Auth.js) — раз БД уже на Supabase, это одна система вместо двух, меньше ручной настройки JWT/сессий.
- Интеграция через Supabase JS SDK прямо в Next.js (Server Actions / API routes) — отдельный backend на Python не нужен, это лишняя сложность без явной причины.
- Не делать собственную систему auth с нуля, потому что в ней много тонкостей: хеширование паролей, сессии, refresh-токены, восстановление доступа, безопасность.

### Оплата

- Stripe

### Деплой

- Vercel — это удобная PaaS-платформа для деплоя Next.js и других веб-приложений: подключаешь GitHub, пушишь код, и платформа сама собирает и разворачивает проект
- Vercel не равен VPS и не заменяет его полностью, но отлично подходит для старта, preview, простых сайтов и API
- Hostinger VPS можно рассматривать позже, когда появится необходимость в собственном сервере, Docker, более сложной инфраструктуре или когда нужно больше контроля

## Почему такой стек

- Next.js + TypeScript + Tailwind — хороший старт для современного сайта.
- Supabase проще, чем развертывать собственный backend с нуля.
- Готовые решения по авторизации и данным ускоряют разработку и снижают риск ошибок.
- Vercel удобен для демонстрации, быстрого деплоя и обучения современному workflow.
- В дальнейшем проект можно масштабировать и добавлять более сложные элементы: Docker, CI/CD, облако, API, админку, аналитики.

## Архитектурный подход

На старте лучше использовать простую и понятную архитектуру:

- Frontend: Next.js App Router
- Content: локальные данные или Supabase
- Admin: простая панель для управления контентом
- Backend: минимальный, по мере необходимости

Это даст хороший баланс между:

- простотой;
- возможностью показать рабочий результат;
- обучением архитектуре;
- постепенным ростом проекта.

## Правила для проекта

- Использовать простые решения там, где это уместно.

- Хранить секреты в переменных окружения.

## Идея на будущее: аналитика/трекинг посетителей

Отслеживание посещаемости и поведения на сайте (сколько заходит, какие страницы смотрят, что кликают).

- **Сделано**: подключён **Vercel Analytics** (`@vercel/analytics`, компонент `<Analytics />` в `src/app/layout.tsx`) — раз сайт и так деплоится на Vercel, самый простой вариант для старта. Показывает посещения страниц, источники трафика, устройства. Данные видны только после деплоя (в дашборде Vercel → Analytics), локально в dev ничего не собирает. Бесплатный лимит — 2500 событий/мес (событие ≈ один просмотр страницы), после лимита сбор данных на месяц останавливается, сайт при этом продолжает работать нормально.
- **Следующий шаг (позже, ориентир — перед Этапом 4/магазином, когда портфолио-сайт уже готов и есть реальные посетители)**: добавить более детальный трекинг поведения — что именно кликают, тепловые карты, записи сессий, воронки покупки. Рассматривать **PostHog** (из коробки даёт heatmaps и session replay, не нужно вручную помечать каждую кнопку событием) как более продвинутую альтернативу/дополнение к Vercel Analytics — особенно ценен на этапе магазина, где важны сценарии "каталог → товар → покупка". Cookie-consent баннер по возможности не усложнять, выбирать privacy-friendly решения (Vercel Analytics/PostHog в базовой конфигурации обычно не требуют баннера).

Если потребуется, этот план можно будет менять по ходу работы.
