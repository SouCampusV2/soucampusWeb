## Текущее состояние сайта

Компактный снимок "что сейчас есть в проекте" — не история изменений (за ней см. `CHANGELOG.md`).

- **Стек:** Next.js 16 (App Router) + TypeScript + Tailwind v4, `framer-motion` (`motion/react`), `@phosphor-icons/react`. Локальный git-репозиторий, коммиты есть.
- **Страницы:** `/` (лендинг), `/portfolio` (каталог), `/portfolio/[slug]` (динамическая деталка проекта), `/reviews/[slug]` (заглушка-подстраница под каждый отзыв, доработать позже — см. `ClientReviews.tsx` → "Read story"), `/about`, `/contact`, `/shop` (живой каталог из БД + `/shop/[slug]` — деталка товара; пока нет опубликованных товаров, показывает пустое состояние и остаётся noindex; покупка на сайте — подэтап B), `/terms` (частично реальная — раздел `#pricing` живой, остальное плейсхолдер до Этапа 4).
- **Компоненты лендинга** (в порядке появления на главной): `Hero`, `Stats`, `RecentProjects`, `CountriesMarquee`, `ClientReviews`, `HowItWorks`, `AuthorCta`, `PricingSection`, `ContactFooter`. Общие для всех страниц: `Navbar`, `PageTransition` (см. `DESIGN.md`), `Skeleton`. `Navbar`, `PageTransition`, `PricingSection` и `ContactFooter` подключены в `src/app/(site)/layout.tsx`, а не в отдельных `page.tsx` — показываются на каждой странице сайта.
- **Роутинг:** все публичные страницы лежат в группе `src/app/(site)/` — скобки означают, что сегмент не попадает в URL (`/about`, а не `/site/about`), он нужен только чтобы у этих страниц был свой общий layout. Корневой `src/app/layout.tsx` держит только то, что нужно вообще всем: `<html>/<body>`, шрифт, `ThemeProvider`, Vercel Analytics. Будущая админка (Этап 3) ляжет в `src/app/admin/` — вне группы `(site)`, поэтому не унаследует навбар/тарифы/футер.
- **Деплой:** прод на `https://soucampus.online` (свой домен, DNS через Hostinger hPanel), автодеплой (CD) через Vercel GitHub App — пуш в `master` пересобирает прод автоматически. CI (GitHub Actions: lint+build) — отдельно, на каждый пуш/PR в `master`. Аналитика — Vercel Analytics (`@vercel/analytics`), базовый бесплатный тариф.
- **Портфолио:** `src/lib/projects.ts` — 16 работ с настоящими фото (`public/portfolio/`), все данные реальные (не плейсхолдер). `/portfolio` — hero-карусель `PortfolioHero.tsx` (первые 5 работ, hover/tap-раскрытие панелей на десктопе, свайп-карусель на мобильном) + bento-сетка "More projects" на остальные. Клик по карточке/панели ведёт на `/portfolio/[slug]` без анимации перехода (`PageTransition` — только на ссылках навбара).
- **Отзывы:** `src/lib/reviews.ts` — 7 реальных отзывов клиентов (`name/role/flag/text/accent/slug`), тот же "table-shaped" паттерн, что и у `projects.ts` — при подключении Supabase (Этап 3) меняется только источник (массив → `fetch` через `getReview(slug)`), сама карточка не трогается. Каждый отзыв кликабелен ("Read story") на свою подстраницу `/reviews/[slug]` — сейчас заглушка (имя/роль/текст), доработать позже. Тот же "table-shaped" подход применён в `Stats.tsx`/`CountriesMarquee.tsx`.
- **Счётчик просмотров (2026-07-20):** считается весь сайт — трекер стоит в `(site)/layout.tsx` и берёт адрес через `usePathname()`, поэтому про новую страницу нельзя забыть. Показывается на `/portfolio`: у каждой работы своё число (`ViewCount.tsx`, бейдж той же высоты, что и тег), плюс общий счётчик уникальных посетителей САЙТА с фильтром периодов (`SiteViews.tsx` — All time / Month / Week / Today; все четыре числа приходят одним серверным запросом, переключение без обращений в сеть). Один человек = единица, сколько бы страниц ни обошёл.
  - **Схема:** строка на тройку (страница, посетитель, день). Ключ именно с днём, иначе периоды невычислимы: при ключе (страница, посетитель) повторный визит не оставляет следа. Плата — "за всё время" считается через `count(distinct visitor_id)`, а не по числу строк.
  - **Почему через браузер:** страницы статические (`revalidate = 60`), серверный код выполняется раз в минуту, а не на каждого гостя — при рендере считались бы пересборки. Сигнал шлёт браузер в `/api/view`.
  - **Доступ:** таблица `page_views` закрыта для анонима полностью, без единой политики. Пишет и читает только сервер под служебным ключом (`supabase-admin.ts`, `SUPABASE_SERVICE_ROLE_KEY` — без префикса `NEXT_PUBLIC_`, поэтому физически не попадает в браузерный бандл).
  - **Защита от накрутки:** cookie подписана HMAC (`visitor-cookie.ts`) — до этого id можно было просто выдумать и тем самым объявить себя "знакомым посетителем", которых лимит не проверяет. Плюс не более 60 РАЗНЫХ посетителей с одного адреса в час (считаются посетители, а не строки: иначе один человек, обошедший 16 работ, блокировал бы следующего гостя с того же NAT). Хэш только от IP, без User-Agent — иначе смена заголовка давала чистый лимит. Сам IP не хранится, только хэш с солью (`VIEW_HASH_SALT`).
  - **Проверка:** юнит-тесты на подпись и белый список адресов + `npm run check:views` — ручной прогон против живого обработчика (подпись, дедупликация, гонка двух вкладок, лимит, смена UA), пишет в настоящую базу с адресов `203.0.113.x` и убирает за собой по `ip_hash` в `finally`. В CI не гоняется намеренно.
  - **На будущее (Этап 4):** когда появятся аккаунты — считать просмотры по пользователям, а не по cookie. Тогда один человек = один аккаунт независимо от браузера, устройства и режима инкогнито, и разовые сбросы вроде смены формата cookie перестанут задваивать людей. Идея пользователя, 2026-07-20.
  - Про cookie и GDPR — вопрос баннера решать вместе с Terms of Service (Этап 4).
- **Тарифы:** `PricingSection` — UI и тарифы/цены финальные, не плейсхолдер. Идея подписки (Patreon-модель) — см. "Идеи на будущее" ниже, пока не реализована.
